Implementacion propuesta (Senior)

Documento de referencia para cumplir el reto al 100% con foco en requisitos mandatorios y evaluacion tecnica.

Objetivo

Entregar una solucion distribuida en 2 microservicios reales, con comunicacion asincrona, Docker y pruebas minimas requeridas, evitando sobre-ingenieria y cumpliendo las reglas de negocio y endpoints exigidos.

Principio rector: cumplir el enunciado exacto antes de agregar complejidad. La evidencia de requisitos mandatorios pesa mas que optimizaciones extra.

Arquitectura propuesta

Microservicio 1

Nombre: ms-clientes-personas

  • Contexto: identidad del cliente.
  • Entidades: Persona, Cliente.
  • Endpoints: /clientes.
  • Publica eventos asincronos: ClienteCreado, ClienteActualizado.

Microservicio 2

Nombre: ms-cuentas-movimientos

  • Contexto: operaciones financieras.
  • Entidades: Cuenta, Movimiento.
  • Endpoints: /cuentas, /movimientos, /reportes.
  • Consume eventos para validar estado del cliente.

Comunicacion: asincrona real via broker (RabbitMQ o Kafka). No usar eventos internos del framework como reemplazo.

Arquitectura sugerida

Descripcion breve de la arquitectura sugerida para el contexto Senior (dos microservicios, mensajeria asincrona y despliegue en contenedores).

Modelo de dominio

Herencia obligatoria

Implementar herencia JPA con Cliente extends Persona.

@Entity
@Inheritance(strategy = InheritanceType.JOINED)
class Persona { ... }

@Entity
class Cliente extends Persona { ... }

Entidades y claves

  • Persona: nombre, genero, edad, identificacion, direccion, telefono.
  • Cliente: clienteId, contrasena, estado.
  • Cuenta: numeroCuenta, tipoCuenta, saldoInicial, estado.
  • Movimiento: fecha, tipoMovimiento, valor, saldo.

Endpoints (CRUD y reportes)

RecursoVerbosDescripcion
/clientesGET, POST, PUT, DELETECRUD de clientes
/cuentasGET, POST, PUT, DELETECRUD de cuentas
/movimientosGET, POST, PUT, DELETECRUD de movimientos
/reportesGETEstado de cuenta por cliente y rango de fechas

Para reportes: /reportes?clienteId=...&fecha=YYYY-MM-DD,YYYY-MM-DD (ajustar formato si se define otro en la API).

Reglas de negocio clave

  • Un movimiento puede ser positivo o negativo.
  • El movimiento actualiza el saldo disponible.
  • Se registra toda transaccion.
  • Si no hay saldo, responder con "Saldo no disponible".
Excepcion de dominio: usar una excepcion especifica (ej. SaldoNoDisponibleException) y mapearla a un error HTTP claro (ej. 400 o 409) con el mensaje exacto.

Comunicacion asincrona

Eventos sugeridos

  • ClienteCreadoEvent con clienteId y estado.
  • ClienteActualizadoEvent cuando cambia estado o datos relevantes.

Flujo minimo

  1. ms-clientes-personas publica evento al crear/actualizar cliente.
  2. ms-cuentas-movimientos consume y actualiza un registro local de clientes habilitados.
  3. Al crear cuenta o movimiento, se valida que el cliente este activo.

Esto demuestra independencia entre servicios y evita acoplarlos via llamadas sincronicas permanentes.

Docker y despliegue

  • Un Dockerfile por microservicio.
  • docker-compose.yml con ambos servicios, base de datos y broker.
  • Variables por servicio: puerto, DB, broker, credenciales.
services:
  ms-clientes:
  ms-cuentas:
  db:
  broker:

Pruebas requeridas

  • 1 prueba unitaria de dominio Cliente (reglas basicas).
  • 1 prueba de integracion (ej. endpoint de movimientos con actualizacion de saldo).

Las pruebas deben ser ejecutables en CI local (maven/gradle) y documentadas en README.

Entregables

  • BaseDatos.sql con esquema y datos base.
  • Coleccion Postman exportada.
  • Repositorio Git publico con instrucciones claras.

Checklist final (antes de entregar)

  • Dos procesos independientes corriendo localmente.
  • Broker en funcionamiento y eventos visibles en logs.
  • Docker build y compose funcionan sin ajustes manuales.
  • Herencia Cliente extends Persona en JPA.
  • Mensaje exacto: "Saldo no disponible".
  • Endpoints probados con Postman y coleccion incluida.
