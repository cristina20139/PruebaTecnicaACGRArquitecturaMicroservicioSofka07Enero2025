Como se comunican los servicios

Respuesta clara sobre el mecanismo de comunicacion y los endpoints de cada microservicio.

Mecanismo de comunicacion

Asincrono por eventos: los servicios no se llaman directamente en el flujo normal. El servicio de clientes publica eventos a un broker (RabbitMQ o Kafka) y el servicio de cuentas los consume.
  • Productor: ms-clientes-personas publica eventos de cliente (creacion/actualizacion).
  • Consumidor: ms-cuentas-movimientos actualiza su vista local del cliente y valida operaciones.
  • Objetivo: desacoplar servicios y cumplir el requisito de microservicios distribuidos.
Evento ejemplo (JSON)
{
  "tipo": "ClienteCreado",
  "clienteId": 101,
  "estado": true
}

Endpoints por servicio

ms-clientes-personas

  • GET /clientes - listar clientes
  • POST /clientes - crear cliente
  • PUT /clientes/{id} - actualizar cliente
  • DELETE /clientes/{id} - eliminar cliente

Eventos publicados: ClienteCreado, ClienteActualizado

ms-cuentas-movimientos

  • GET /cuentas - listar cuentas
  • POST /cuentas - crear cuenta
  • PUT /cuentas/{id} - actualizar cuenta
  • DELETE /cuentas/{id} - eliminar cuenta
  • GET /movimientos - listar movimientos
  • POST /movimientos - crear movimiento
  • PUT /movimientos/{id} - actualizar movimiento
  • DELETE /movimientos/{id} - eliminar movimiento
  • GET /reportes?clienteId=...&fecha=YYYY-MM-DD,YYYY-MM-DD
